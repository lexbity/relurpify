# Workspace Layout

## Synopsis

`relurpify_cfg/` is the only workspace-local source of truth for Relurpify runtime configuration and generated state. Shared templates are seed material only. Once copied into a workspace, the files inside `relurpify_cfg/` become authoritative for `relurpish`, `dev-agent`, and Relurpify apps.

---

## Workspace Contract

Canonical workspace layout:

```text
relurpify_cfg/
├── config.yaml
├── agent.manifest.yaml
├── agents/
├── skills/
├── logs/
├── telemetry/
├── memory/
├── sessions/
└── test_runs/
```

### Declarative, Workspace-Owned Files

- `config.yaml` stores workspace defaults such as preferred model, agent, and temporary tool narrowing.
- `agent.manifest.yaml` is the active default manifest for the workspace.
- `agents/` contains additional manifests copied into and owned by the workspace.
- `skills/` contains skill packages copied into and owned by the workspace.

These files are the runtime source of truth once they exist in the workspace.

### Generated Runtime State

- `logs/` stores runtime logs such as `relurpish.log`.
- `telemetry/` stores structured telemetry output such as `telemetry.jsonl`.
- `memory/` stores memory/index state.
- `sessions/` stores persisted sessions and workflow state.
- `test_runs/` stores isolated testsuite inputs and outputs.

These paths are generated by runtime use. They should generally not be treated as starter assets.

---

## Template Ownership

Relurpify now distinguishes between:

- shared installed templates
- repo-local development fallback templates
- workspace-owned runtime copies

Shared template roots are machine-local and resolved through the template resolver. They are not read as live runtime state. `relurpish doctor` and testsuite derivation copy these templates into workspace-local `relurpify_cfg/` trees.

The ownership rule is simple:

- template files are copy-from only
- workspace copies are editable and authoritative

---

## Doctor Initialization

`relurpish doctor` is the supported workspace setup flow.

If `relurpify_cfg/` is missing, doctor can:

- create the workspace config root
- copy starter `config.yaml`
- copy starter `agent.manifest.yaml`
- create expected runtime directories
- validate Docker, `runsc`, Ollama, and Chromium

`doctor --fix` can overwrite the starter workspace files from current templates. That is intentional: the copied workspace files remain authoritative after the overwrite.

---

## Testsuite Layout

Tests run in isolated, derived workspaces under the live workspace tree:

```text
relurpify_cfg/test_runs/<agent>/<run-id>/
├── tmp/
│   └── <case>/
│       └── workspace/
│           └── relurpify_cfg/
├── logs/
├── telemetry/
├── artifacts/
└── report.json
```

The temp workspace is derived from a testsuite template profile and layered with suite or case overrides. Tests must not write into the live workspace's top-level `logs/`, `telemetry/`, `memory/`, or `sessions/` directories.

---

## Tool Split

- `relurpish` is the end-user runtime and TUI.
- `relurpish doctor` is the supported workspace initialization and local dependency check flow.
- `dev-agent` is the development and integration CLI.

The workspace contract is shared, but the tools serve different audiences.
