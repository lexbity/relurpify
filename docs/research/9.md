# Euclo-Owned Relurpic Capabilities

## Synopsis

Relurpic capability is now explicitly documented as a framework primitive
through `CapabilityRuntimeFamilyRelurpic`.

This document defines the next layer above that primitive:

- `framework/` owns the relurpic capability concept
- `named/euclo/` owns Euclo's coding-specific relurpic capability families

Euclo should therefore be understood as:

- a UX-agnostic named coding runtime
- that orchestrates `UnitOfWork`
- across multiple software-development modes
- by selecting and running Euclo-owned relurpic capabilities
- which may compose tools, skills, sub-agents, and multiple execution paradigms

This is the intended runtime contract for Euclo going forward.

---

## Core Model

### Layering

- `framework/`
  - owns capability primitives, including relurpic runtime-family semantics
  - owns skill resolution, capability admission, policy, context, restore, and
    execution substrate contracts
- `agents/`
  - owns reusable generic execution paradigms
  - may also own generic relurpic helpers and reusable generic relurpic
    behaviors
- `archaeo/`
  - owns memory, provenance, living-plan state, and codebase-knowledge
    relationships
- `named/euclo/`
  - owns Euclo modes
  - owns Euclo `UnitOfWork`
  - owns Euclo-specific relurpic capabilities
  - owns runtime guarantees, deferrals, continuity, and final execution results

### Runtime Roles

- Archaeo requests, stores, and exposes semantic memory/provenance state.
- Relurpic capabilities reason and execute coding-specific behavior assemblies.
- Euclo orchestrates execution, transitions, policy, continuity, and final
  outcome semantics.

This means Euclo should not treat relurpic capability as an incidental helper.
It should treat relurpic capability as the primary behavior layer it owns.

---

## Euclo Modes

Euclo modes are UX-agnostic software-development paradigms. They are not merely
menu states.

The current authoritative modes are:

- `chat`
- `archaeology`
- `plan`
- `debug`

For current architecture purposes, `archaeology` and `plan` are the same
planning-oriented mode family from Euclo's perspective, even if UX surfaces may
present them differently.

### Shared Context

Modes share context, memory, and session continuity where possible.

Mode transitions may preserve the same `UnitOfWork` when the execution context
is still semantically continuous. However, transitions from an Archaeo-backed
memory/provenance path into a local-only path are allowed to start a new
`UnitOfWork` when that produces a clearer execution contract.

---

## Euclo-Owned Relurpic Capability Model

Each mode must have:

- one required primary behavior owner
- zero or more supporting relurpic capabilities

The primary owner is responsible for:

- owning the behavior assembly for that mode invocation
- coordinating supporting relurpic capabilities
- staying within Euclo runtime policy, security, and deferral guarantees

Supporting relurpic capabilities may contribute:

- specific reasoning routines
- sub-agent orchestration
- paradigm composition
- tool capability use
- verification and evidence gathering

### Capability Composition Rule

A Euclo relurpic capability may compose:

- framework capabilities
- framework skills
- one or more `/agents` paradigms
- one or more sub-agents
- Archaeo-associated semantic input acquisition

This is expected and intentional.

### Security Rule

Relurpic capabilities must not bypass:

- capability admission
- manifest-driven permissions
- skill-constrained capability sets
- tool capability security boundaries

Tool use remains a capability-layer concern even when the capability is wrapped
inside a relurpic behavior.

## Capability Catalog

### Chat Mode

`chat` is the broad execution surface. It is not one narrow behavior.

Euclo-owned relurpic capabilities:

- `euclo:chat.ask`
- `euclo:chat.implement`
- `euclo:chat.inspect`

#### `euclo:chat.ask`

Purpose:

- answer engineering questions
- explain code, architecture, or data flow
- surface reasoning through shared context and available evidence

Guarantees:

- non-mutating

May use:

- local code/context collection
- non-mutating tools and capabilities
- optionally lazy Archaeo-backed semantic input acquisition if the request
  requires wider codebase grounding

#### `euclo:chat.implement`

Purpose:

- perform direct coding/edit execution
- collect local context and use toolbox-style execution
- support standard coding-agent style behavior when the user asks for code work

Guarantees:

- mutating execution is allowed when admitted by policy

May use:

- ReAct-style tool/capability collection
- supporting verification and repair behaviors
- lazy transition into Archaeo-backed context when the task proves long-horizon
  enough

#### `euclo:chat.inspect`

Purpose:

- inspect code, state, logs, files, or behavior without being framed primarily
  as a direct user question

Guarantees:

- not guaranteed non-mutating, because available tool capabilities may not
  guarantee that
- however, the intended posture is inspect-first and policy-constrained

May use:

- tool capability output
- local codebase inspection
- selective evidence gathering

### Archaeology / Plan Mode

Any LLM-dependent archaeology behavior should be represented as an
Archaeo-associated Euclo relurpic capability.

Euclo-owned relurpic capabilities:

- `euclo:archaeology.explore`
- `euclo:archaeology.compile-plan`
- `euclo:archaeology.implement-plan`

#### `euclo:archaeology.explore`

Purpose:

- explore codebase structure, provenance, behavior, ideas, tensions, and
  pattern-bearing code relationships
- support both scope widening and scope cutting

Association:

- Archaeo-associated

May compose:

- pattern-recognition routines
- prospective reasoning
- provenance review
- convergence and coherence analysis

#### `euclo:archaeology.compile-plan`

Purpose:

- turn archaeology-backed exploration and accepted semantic state into a full
  executable plan

Association:

- Archaeo-associated

Guarantees:

- emits a full compiled plan suitable for Euclo execution
- does not emit partial plan outputs as successful completion

Failure handling:

- issues during plan compilation should become deferred artifacts for the end
  user to resolve later

#### `euclo:archaeology.implement-plan`

Purpose:

- execute against a compiled living plan
- support long-horizon implementation, including TDD-style implementation where
  applicable

Association:

- Archaeo-associated

Guarantees:

- requires a compiled/finalized living plan before execution begins
- executes under Euclo's single-plan-per-run rule

### Debug Mode

Euclo-owned relurpic capability:

- `euclo:debug.investigate`

This is the required primary owner for debug execution.

It may internally compose:

- root-cause investigation
- defect hypothesis refinement
- verification-driven localization
- flaw / smell / vulnerability surfacing
- targeted verification repair
- tool output exposition
- Archaeo-backed memory/provenance reads when useful

`tool exposition` is a required facet of `debug.investigate`, not a separate
standalone capability. This prevents bypassing the current capability/security
model for tool access.

Mutation rule:

- if debug execution needs to cross into repair/implementation beyond its
  bounded investigation contract, it should transition to `euclo:chat.implement`

## Archaeo-Associated Vs Euclo-Local

### Archaeo-Associated

A Euclo relurpic capability is Archaeo-associated when it relies on:

- living-plan state
- provenance
- codebase memory
- semantic knowledge relationships
- Archaeo-backed pattern/prospective/convergence routines

These capabilities still belong to Euclo if they are part of Euclo's coding
runtime, but they integrate with Archaeo by design.

### Euclo-Local

A Euclo relurpic capability is Euclo-local when it does not require Archaeo to
operate.

These capabilities may still rely on:

- framework capabilities
- framework skills
- context management
- provider restore
- `/agents` paradigms

but they do not depend on Archaeo-backed semantic memory/provenance.

### Lazy Acquisition

Some capabilities may lazily acquire Archaeo-backed semantic inputs during
execution. This is valid when:

- the capability begins in a mostly local posture
- wider codebase grounding becomes necessary
- policy and runtime context still allow a continuous execution path

## UnitOfWork Integration

`UnitOfWork` should bind:

- one primary Euclo relurpic capability owner
- zero or more supporting relurpic capabilities
- execution policy
- context/runtime policy
- optional Archaeo semantic bundle
- capability/security constraints

This is more precise than treating executor selection as the top-level decision.

The intended order is:

1. resolve mode
2. build or continue `UnitOfWork`
3. select primary Euclo relurpic capability
4. bind supporting relurpic capabilities
5. materialize paradigm/sub-agent/tool composition under that capability
6. execute under Euclo runtime guarantees

---

## Transition Rules

### Same `UnitOfWork`

Mode transitions should preserve the same `UnitOfWork` when:

- the semantic objective is still continuous
- security and context policy remain compatible
- there is no sharp shift from Archaeo-backed to local-only execution

### New `UnitOfWork`

A new `UnitOfWork` is allowed when:

- execution leaves Archaeo-backed memory/provenance context for a local-only
  path
- the transition changes the primary execution contract enough that continuity
  would become misleading

Transitions should remain linked through artifacts and runtime references even
when a new `UnitOfWork` is created.

---

## Naming And Versioning

Euclo-owned relurpic capabilities should use stable semantic IDs.

Current IDs:

- `euclo:chat.ask`
- `euclo:chat.implement`
- `euclo:chat.inspect`
- `euclo:archaeology.explore`
- `euclo:archaeology.compile-plan`
- `euclo:archaeology.implement-plan`
- `euclo:debug.investigate`

Cross-mode reuse should remain implicit for now.

Deferred design note:

- future tagging may be needed so capabilities can advertise reuse,
  interoperability, and mode transition compatibility without introducing a
  separate shared namespace yet

---

## Engineering Consequences

The main implementation consequences are:

- Euclo should own its relurpic capability registry in `named/euclo`
- `/agents/relurpic` should remain for generic relurpic helpers and reusable
  generic behavior, not as the owner of Euclo's coding capability catalog
- executor selection should become subordinate to relurpic capability selection
- tests should assert relurpic capability ownership and guarantees by mode
- security and capability admission should remain capability-first even when
  behavior assembly composes tools and sub-agents

---

## Full-Product Direction

This model is intended for full implementation, not a minimal subset.

The next work should therefore define:

- the Euclo relurpic capability registry and descriptors
- per-capability contracts and metadata
- supporting capability sets per mode
- transition rules and `UnitOfWork` rebinding
- tests and benchmarks per mode/capability family

This is the path toward Euclo being the clear coding-runtime owner while still
respecting:

- `framework/` as primitive/policy owner
- `agents/` as generic paradigm substrate
- `archaeo/` as memory/provenance/knowledge owner
