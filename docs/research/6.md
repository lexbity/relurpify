# Euclo Single-Plan Execution And Deferral Specification

Status

Proposed full-product engineering specification.

Scope

This document refines [5.md](/home/lex/Public/Relurpify/docs/research/5.md) into a narrower runtime contract for:

- single-plan execution per run
- deferred execution issue semantics
- execution result classes
- deferral artifact structure
- behavior-oriented relurpic routine families

This is a Euclo runtime specification, not a UX specification.

## 1. Core Decision

For a single Euclo execution run, there is exactly one fully formed plan version.

Euclo does not create successor plans during execution.

Execution-time ambiguity, stale assumptions, pattern tensions, non-fatal failures, verification concerns, and provider constraints do not mutate the active plan in place and do not automatically create successor draft plans.

Instead, Euclo records them as deferred execution issues and continues where safe.

Replanning is a later explicit archaeology activity initiated by the end user.

## 2. Execution Contract

### 2.1 Single-Plan Binding

Each execution run must bind to exactly:

- one `workflow_id`
- one `run_id`
- one compiled execution contract
- one active plan version

That binding remains stable for the life of the run.

### 2.2 No In-Run Plan Branching

The active plan version must not be replaced, forked, or superseded automatically during the run.

If execution discovers new knowledge that may affect future planning, Euclo records a deferred execution issue rather than mutating the plan model for the current run.

### 2.3 Replanning Boundary

Replanning is outside the current execution run.

It begins only when:

- the end user decides to revisit deferred issues
- archaeology is explicitly re-entered
- a new plan is formed for a subsequent run

### 2.4 Step Validity

Step validity is not recomputed into a replacement plan during the active run.

Conservative recomputation belongs to the next plan adoption cycle, after the current run has completed and archaeology has been revisited.

## 3. Why This Model Exists

This model is intentionally stricter than a dynamic replan-in-place system.

It exists to preserve:

- orchestration clarity
- end-user investment in plan formation
- run stability
- explicitness of discovered execution-time knowledge
- clean separation between execution and planning

The key product rule is:

one run, one plan, many possible deferrals.

## 4. Deferred Execution Issues

Deferred execution issues are first-class Euclo runtime artifacts.

They are not merely logs, warnings, or UI events.

They capture unresolved execution-time knowledge that must survive the run and remain available for later user-driven archaeology re-entry.

### 4.1 Definition

A deferred execution issue is a Euclo-owned runtime object created when execution encounters a meaningful concern that:

- is real enough to preserve
- cannot or should not be fully resolved inline
- does not justify stopping the run immediately

### 4.2 Deferred Execution Issue Taxonomy

The baseline Euclo taxonomy is:

- `ambiguity`
- `stale_assumption`
- `pattern_tension`
- `nonfatal_failure`
- `verification_concern`
- `provider_constraint`

If Archaeo exposes a compatible taxonomy, Euclo should align to it or map to it.

If Archaeo does not yet provide a compatible taxonomy, Euclo must still expose this stable execution-oriented taxonomy.

### 4.3 Creation Rules

Deferred execution issues should be created when execution encounters:

- unresolved ambiguity
- stale assumptions discovered during implementation
- pattern tensions that remain unresolved during the run
- failures that do not require hard stop
- verification concerns that do not invalidate the entire run
- provider limitations or temporary constraints that affect confidence or completeness without making continuation unsafe

### 4.4 Non-Goals

Deferred execution issues are not:

- automatic successor plans
- implicit plan invalidations
- replacements for explicit hard failures
- UX-only notifications

## 5. Scope Model For Deferrals

Deferred execution issues must exist at both:

- step scope
- run scope

### 5.1 Step Scope

Every issue should record:

- the active step that discovered it
- any related step identifiers
- the local execution context that produced it

### 5.2 Run Scope

Every issue must also be visible as a run-level artifact so that:

- final execution reporting can include it
- later archaeology re-entry can discover it without replaying the whole run
- downstream UX layers can notify the user consistently

## 6. Required Deferred Issue Fields

Each deferred execution issue must include at minimum:

- `issue_id`
- `workflow_id`
- `run_id`
- `execution_id`
- `active_plan_id`
- `active_plan_version`
- `step_id`
- `related_step_ids`
- `kind`
- `severity`
- `status`
- `title`
- `summary`
- `created_at`
- `updated_at`
- `why_not_resolved_inline`
- `recommended_reentry`
- `recommended_next_action`

### 6.1 Evidence Payload

Evidence is essential.

Each issue must carry enough supporting evidence for later exploration if the user pursues it.

Minimum evidence fields should include:

- `touched_symbols`
- `relevant_pattern_refs`
- `relevant_tension_refs`
- `relevant_anchor_refs`
- `relevant_provenance_refs`
- `relevant_request_refs`
- `verification_refs`
- `checkpoint_refs`
- `provider_state_snapshot`
- `short_reasoning_summary`

### 6.2 Archaeo References

Each issue should include references to related Archaeo records when available, such as:

- tensions
- learning interactions
- requests
- provenance entries
- phase or timeline events

Euclo owns the issue.

Archaeo references provide the semantic trail.

## 7. Deferral Artifact Persistence

Every deferred execution issue must be persisted in two forms:

- workflow-scoped Euclo runtime artifact
- workspace-visible artifact for end-user review

### 7.1 Workspace Artifact Format

The workspace-visible artifact format is:

- Markdown body
- YAML frontmatter

This allows:

- human readability
- stable filesystem references
- easy future parsing by tools

### 7.2 Markdown Artifact Requirements

The artifact should contain:

- a concise issue title
- the issue kind and severity
- the active plan and step references
- a short explanation of the issue
- why execution continued
- what evidence was captured
- relevant Archaeo references
- recommended re-entry path

### 7.3 YAML Frontmatter Requirements

Suggested required frontmatter fields:

```yaml
issue_id: "defer-..."
workflow_id: "wf-..."
run_id: "run-..."
execution_id: "exec-..."
plan_id: "plan-..."
plan_version: 3
step_id: "step-7"
kind: "pattern_tension"
severity: "medium"
status: "open"
created_at: "2026-03-27T00:00:00Z"
recommended_reentry: "archaeology"
archaeo_refs:
  tensions: ["tension-1"]
  provenance: ["prov-9"]
```

The exact path convention may be finalized later, but the runtime contract requires these artifacts to be durable and referenceable.

## 8. Execution Result Classes

Execution outcomes must be represented as top-level result classes, not just a success boolean with attached metadata.

### 8.1 Required Result Classes

Euclo should expose at minimum:

- `completed`
- `completed_with_deferrals`
- `blocked`
- `failed`
- `canceled`
- `restore_failed`

### 8.2 Completed With Deferrals

`completed_with_deferrals` is a distinct top-level result class.

It means:

- the run completed sufficiently to be considered operationally successful
- one or more deferred execution issues remain open
- the end user must be informed that discovered knowledge remains unresolved

This is not equivalent to plain success.

### 8.3 Relationship To Success Gates

Verification and execution success gates still matter.

However, the final classification should distinguish:

- completed without unresolved execution concerns
- completed with unresolved execution concerns

That distinction must survive into artifacts and UX consumers.

## 9. Continuation Rules

Euclo should continue execution by default when a deferred issue is created, unless continuation would become unsafe.

### 9.1 Continue Cases

Execution should generally continue when the issue is:

- informative but unresolved
- confidence-reducing but not invalidating
- semantically important but not blocking
- operationally inconvenient but not safety-breaking

### 9.2 Hard Stop Cases

Execution should still stop for:

- explicit user cancellation
- irrecoverable restoration failure
- irrecoverable sandbox/security denial
- unrecoverable runtime corruption
- execution states that are unsafe to continue

### 9.3 Why Continue Matters

The product requirement is that long-running implementation should not collapse into repeated interruptions for every unresolved concern.

Deferred execution issues are the mechanism that allows progress without loss of knowledge.

## 10. Deferral Lifecycle

Suggested lifecycle states:

- `open`
- `acknowledged`
- `resolved`
- `ignored`
- `superseded`
- `reentered_archaeology`

### 10.1 In-Run Lifecycle

During the run, issues will generally be created as `open`.

They may be updated with additional evidence if the same concern recurs.

### 10.2 Post-Run Lifecycle

After the run:

- the end user may review them
- archaeology may be re-entered
- a subsequent plan/run may supersede or resolve them

## 11. Compiled Execution Relationship

The compiled execution artifact must contain references to all deferred execution issues created during the run.

At minimum it should track:

- open issue IDs
- issue counts by taxonomy
- latest issue timestamps
- whether completion class is `completed_with_deferrals`

This allows context restoration and final reporting to remain accurate without re-scanning all artifacts.

## 12. Behavior-Oriented Relurpic Routine Families

Relurpic capabilities should be organized primarily by execution behavior.

This is closer to their architectural role than user-facing scenario labels.

### 12.1 Initial Core Routine Families

The initial strongly defined routine families should include:

- `gap_analysis`
- `verification_repair`
- `scope_expansion_assessment`
- `stale_assumption_detection`
- `tension_assessment`

Some of these may already exist in partial or concrete form. This specification formalizes them as core execution behavior families.

### 12.2 Routine Family Purpose

These routines exist to provide focused reasoning inside execution without forcing all reasoning into one generic loop.

They should operate as:

- reusable execution behaviors
- selectable by mode and step context
- composable with capability/tool policies

### 12.3 Not User-Facing By Default

Relurpic routine families are not primarily UX labels.

They are runtime execution behaviors that Euclo uses internally or exposes indirectly through higher-level controls.

## 13. Routine Selection Contract

Routine selection should be driven by:

- mode
- active phase
- active plan step
- compiled execution state
- current issue taxonomy
- verification state

### 13.1 Examples

- `gap_analysis` when a step completes but semantic consistency is uncertain
- `verification_repair` when verification produced actionable failure signals
- `scope_expansion_assessment` when observed blast radius exceeds expected scope
- `stale_assumption_detection` when retrieved or plan-bound assumptions appear outdated
- `tension_assessment` when touched symbols intersect known tensions or pattern conflicts

### 13.2 Product Direction

Euclo should rely more on these narrowly defined execution behaviors and less on broad undifferentiated reasoning loops.

This is especially important for lower-end models.

## 14. Relationship To Archaeo

Archaeo remains the direct integration target.

Euclo and relurpic capabilities should use `/archaeo` directly through services and bindings, not through GraphQL as the default path.

### 14.1 Archaeo Responsibilities

Archaeo continues to own:

- requests
- evaluations
- living plans
- provenance
- tensions
- learning state
- deferred drafts

### 14.2 Euclo Responsibilities

Euclo continues to own:

- execution
- execution artifacts
- deferred execution issues
- completion classes
- context continuation

### 14.3 Important Boundary

Any remaining inline provider path in Archaeo should be treated as transitional compatibility, not target architecture.

The intended architecture is:

- Archaeo requests and evaluates
- Euclo executes
- relurpic capabilities reason

## 15. Relationship To Archaeo Deferred Drafts

Deferred execution issues and Archaeo deferred drafts must remain conceptually distinct.

### 15.1 Deferred Execution Issue

Euclo runtime concern:

- discovered during execution
- tied to step and run
- may be non-fatal
- intended for later user review and possible archaeology re-entry

### 15.2 Deferred Draft

Archaeo planning/provenance concern:

- part of archaeology-side semantic/planning workflows
- not automatically spawned by the execution run

### 15.3 Contract

Euclo should reference Archaeo deferred or related records when they exist, but must not treat its own deferred execution issues as implicit plan branches.

## 16. Lower-End Model Implications

This execution contract is specifically favorable for lower-end models because it:

- reduces in-run replanning complexity
- favors narrow execution behaviors
- keeps planning and execution cleanly separated
- preserves unresolved knowledge without demanding inline full resolution
- supports restoration and continuation without reconstructing the full run from prose context

## 17. Required Test Scenarios

The following scenarios should be treated as core runtime tests for this model:

- run completes with no deferrals
- run completes with one step-scoped deferral
- run completes with multiple run-scoped deferrals across steps
- run records verification concern but still returns `completed_with_deferrals`
- run records stale assumption and preserves evidence for later archaeology re-entry
- run restores context mid-execution and preserves deferred issue visibility
- subsequent archaeology re-entry can discover deferred issue artifacts without replaying the original run

## 18. Open Integration Questions

This document leaves some implementation details open:

- exact artifact directory convention
- exact severity scale
- exact mapping between Euclo issue taxonomy and Archaeo record taxonomy
- exact wire format for compiled execution references to issue artifacts

These do not change the contract defined here.

## 19. Summary

The key decisions in this specification are:

- one fully formed plan per execution run
- no automatic successor plan creation during execution
- unresolved execution-time concerns become deferred execution issues
- deferred issues are both step-scoped and run-scoped
- deferred issues persist as workflow artifacts and markdown-plus-YAML workspace artifacts
- `completed_with_deferrals` is a top-level result class
- relurpic capabilities should be organized around execution behaviors
- Euclo integrates directly with Archaeo, while GraphQL remains an optional transport surface
