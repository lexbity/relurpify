# Euclo Full-Product Engineering Specification

Status

Proposed full-product engineering specification.

Scope

This specification defines Euclo as a UX-agnostic coding runtime built on top of:

- `/framework` for context management, memory, orchestration, capability plumbing, and sandbox-aware execution
- `/agents` for reusable agent paradigms and relurpic execution routines
- `/archaeo` for living planning, workspace memory, provenance, restoration, and external data-plane access

This document is not a Relurpish UX specification. Relurpish is one UX over Euclo. Euclo must remain usable by other UX layers and transports.

This document is also not an Archaeo UX specification. Archaeo remains data-driven and externally exposable through surfaces such as `/app/archaeo-graphql-server`.

## 1. Product Intent

Euclo is the coding-focused runtime in Relurpify.

Its purpose is:

- to execute software engineering work reliably
- to remain effective when using lower-end LLMs
- to operate over long-running workflow-backed implementation tasks
- to preserve continuity through context clearing, compaction, and restoration
- to expose LLM capability as grounded knowledge and execution support rather than opaque prose

Euclo is not the owner of workspace memory or provenance. It consumes those from Archaeo.

Archaeo is not the owner of coding execution. It supplies the living plan, workspace memory, provenance, deferred records, tensions, and restoration substrate that Euclo executes against.

## 2. Layering Contract

### 2.1 Framework

`/framework` owns:

- context and memory primitives
- workflow/runtime storage
- agent orchestration
- capability/tool registration
- approvals and sandbox integration
- skills/configuration substrate

`/framework` does not own coding-specific execution semantics.

### 2.2 Agents

`/agents` owns:

- generic agent paradigms
- relurpic routines and capability families
- reusable execution behaviors
- specialist implementations that Euclo may invoke

`/agents` does not define Eucloâ€™s runtime contract.

### 2.3 Archaeo

`/archaeo` owns:

- living plans
- workspace memory and provenance
- exploration snapshots
- tensions, learning queues, deferred drafts, convergence, decisions
- request lifecycle and external fulfillment boundaries
- restoration substrate for long-running continuity

Archaeo is the semantic and planning authority.

### 2.4 Euclo

`/named/euclo` owns:

- coding-task execution semantics
- mode/phase-driven coding runtime behavior
- compiled execution state
- execution artifact lifecycle
- deferred execution issue lifecycle
- context compaction and restore coordination
- step execution over a living plan
- verification, checkpoint, and recovery policy for coding tasks

Euclo remains UX agnostic.

### 2.5 Relurpish

`/app/relurpish` owns:

- TUI behavior
- pane/tab affordances
- human-visible status and notifications
- mode-specific interaction presentation

Relurpish must consume Euclo runtime state rather than define it.

## 3. Primary Execution Model

Euclo remains mode/phase driven.

Long-running coding workflows are expected to be started from archaeology-backed planning, but once execution begins, Euclo remains the runtime driving the implementation loop.

The primary control structure is:

`mode -> phase -> execute`

where phase transitions are informed by:

- the living plan from Archaeo
- compiled execution state
- verification output
- relurpic routine selection
- deferred execution issue policy
- context lifecycle state

This is intentionally different from making Archaeo the execution loop. Archaeo provides the live plan and memory substrate. Euclo executes against it.

## 4. Workflow-Backed Execution

All substantial coding tasks should be workflow-backed by default.

A task is considered substantial when it involves one or more of:

- multi-step implementation
- long-running execution
- plan-backed work
- archaeology-backed context restoration
- deferred execution issue tracking
- provider/session degradation tolerance
- stepwise verification and checkpointing

Non-workflow-backed execution may still exist for lightweight tasks, but the full-product path assumes workflow-backed execution is the default for serious coding work.

## 5. Runtime Goals

Eucloâ€™s full-product runtime must optimize for:

- reliable coding execution
- strong continuity across sessions and compaction
- explicit tracking of uncertainty and unresolved issues
- effective operation on lower-end models
- grounded knowledge surfacing through patterns, tensions, provenance, and plan context
- reuse of relurpic routines as execution presets and specialist behaviors

## 6. Hard Guarantees

Euclo should uphold the following hard guarantees for workflow-backed long-running implementation tasks:

- no silent loss of plan intent
- no silent loss of workflow continuity
- no silent mutation completion without surfaced verification state
- no silent deferred execution issues
- context clearing or compaction must not stop long-running implementation tasks
- automatic context restore must be available for long-running implementation tasks using Archaeo-backed living plans
- restoration must preserve enough intent to continue execution safely

These are product guarantees, not minimum viable behavior.

## 7. Operational Regimes

Euclo has two broad operational regimes.

### 7.1 Exploration and Planning Regime

Characteristics:

- archaeology-heavy
- pattern and tension aware
- user-facing reasoning support
- more likely to use richer relurpic routines
- creates or refines the living plan

### 7.2 Long-Running Execution Regime

Characteristics:

- executes against a compiled representation of the living plan
- keeps moving unless explicitly canceled or hard-blocked
- tolerates context compaction and provider degradation
- records deferred execution issues instead of repeatedly stopping
- uses verification and checkpoints to keep progress grounded

The minimum top-level runtime state categories are:

- execution identity

## 9. Compiled Execution Artifact

Compiled execution state is a first-class Euclo artifact.

It exists so that long-running execution does not depend on:

- current chat history
- current in-memory context
- re-deriving execution state from scratch after every interruption

### 9.1 Artifact Purpose

The compiled execution artifact is the Euclo-owned machine-readable contract for continuing work against the current living plan.

It is the primary continuity anchor for:

- resume
- restore after compaction
- step selection
- relurpic routine selection
- verification expectations
- provider degradation handling

### 9.3 Human Readability

The compiled execution artifact should be primarily optimized for machine continuity.

A human-readable companion summary should also be emitted so the end user can inspect:

- what Euclo thinks it is doing
- what step is active
- what assumptions are in force
- what verification is expected
- what unresolved issues exist

### 9.4 Persistence

Compiled execution artifacts must be persisted as workflow-scoped Euclo artifacts.

They should also be resumable without requiring the original execution context window.

### 9.5 Rebuild and Invalidation

Compiled execution state must be rebuildable from:

- the current living plan
- execution history
- checkpoints
- Archaeo provenance

It must be invalidated or recompiled when:

- the active plan version changes incompatibly
- the active step becomes invalidated
- anchor dependencies drift beyond tolerance
- restoration cannot satisfy continuation requirements
- a mode change requires a new execution preset

## 10. Deferred Execution Issues

Deferred execution issues are first-class Euclo objects and artifacts.

They are not just UI notifications and not just Archaeo drafts.

They represent unresolved execution-time concerns that did not justify stopping the long-running implementation task.

### 10.1 Purpose

Deferred execution issues allow Euclo to continue implementation while preserving unresolved concerns for later evaluation.

### 10.2 Creation Triggers

Deferred execution issues should be created for at least:

- stale assumptions
- ambiguity that does not require immediate stop
- non-fatal execution failures
- unresolved pattern tension

They may also be created for:

- degraded provider conditions
- partial verification contradictions below hard-stop thresholds
- recovery paths that succeeded but left material uncertainty

### 10.3 Artifact Requirements

Every deferred execution issue must produce:

- a workflow-scoped Euclo artifact
- a human-referenceable workspace artifact
- a reference to the associated Archaeo record when one exists

The workspace artifact should be left for the end user to reevaluate or use as a re-entry point into archaeology.

### 10.4 Required Fields

Each deferred execution issue must include at minimum:

- `issue_id`
- `workflow_id`
- `run_id`
- `execution_id`
- `created_at`
- `updated_at`
- `status`
- `severity`
- `title`
- `summary`
- `issue_kind`
- `active_step_id`
- `related_step_ids`
- `related_pattern_refs`
- `related_tension_refs`
- `related_request_refs`
- `related_provenance_refs`
- `related_archaeo_record_refs`
- `trigger_reason`
- `evidence`
- `recommended_reentry`
- `recommended_next_action`
- `workspace_artifact_path`

### 10.5 Lifecycle States

Suggested states:

- `open`
- `acknowledged`
- `superseded`
- `reentered_archaeology`
- `resolved`
- `ignored`

### 10.6 Workspace Artifact Form

The artifact should be durable, end-user readable, and easy to link to from UX layers.

Suggested properties:

- text or markdown form
- stable identifier
- explicit Archaeo references
- explicit note that execution continued despite the issue

Directory conventions may be finalized later, but the runtime contract requires these artifacts to exist.

## 11. Context Compaction and Automatic Restore

Context management must be treated as a normal part of long-running execution, not as a terminal failure.

### 11.1 Design Goal

Context compaction and clearing must not stop a long-running implementation task when:

- the workflow is intact
- the living plan is intact
- sufficient execution artifacts and provenance exist to restore continuation safely

### 11.2 Context Lifecycle States

Required states:

- `full`
- `degraded`
- `compacted`
- `restoring`
- `restored`
- `restore_failed`

### 11.3 Restore Contract

Automatic restoration must:

- recover the active step and surrounding dependencies
- recover execution intent
- recover unresolved deferred execution issues
- recover verification expectations
- recover enough provenance to continue safely

Automatic restoration does not need to reconstruct the entire original context window verbatim.

It must reconstruct enough of the execution narrative to preserve safe continuation.

### 11.4 Restore Inputs

Restore may use:

- compiled execution artifact
- active living plan version
- Archaeo projections
- provenance and timeline
- checkpoint summaries
- deferred execution issue artifacts
- step history

### 11.5 Restore Failures

Restore should only hard-block execution when continuation cannot be made safe.

Examples include:

- missing or corrupt workflow state
- missing or invalid living plan
- incompatible plan changes without recompilation
- irrecoverable checkpoint mismatch

## 12. Verification and Checkpoints

Verification remains a core Euclo responsibility for coding execution.

### 12.1 Verification State

Euclo must persist verification state as a first-class artifact and runtime state surface.

Required fields include:

- verification policy
- required checks
- observed checks
- verification status
- unresolved contradictions
- verification timestamp

### 12.2 Checkpoint Semantics

Checkpoints are not just raw tool results.

They are semantic execution markers tied to:

- active step
- plan version
- verification state
- relevant anchors or provenance
- recovery position

### 12.3 Checkpoint Uses

Checkpoints support:

- resume
- restore
- rollback
- auditability
- verification lineage

## 13. Relurpic Routine Model

Relurpic capabilities are execution routines and presets.

They are not primarily a user-visible UX concept.

### 13.1 Selection Model

Routine selection is determined by:

- mode
- active step characteristics
- compiled execution state
- living plan context
- recovery and verification state

This is a hybrid of:

- preset-driven behavior
- step-driven specialization

### 13.2 Routine Sources

Relurpic routines may encapsulate:

- configuration presets
- `/framework/skills`
- reusable `/agents` implementations
- scenario-specific execution strategies

### 13.3 Product Requirement

Euclo must be able to select and run specialized relurpic routines without exposing those routines as mandatory end-user UX concepts.

## 14. LLM Availability and Lower-End Models

Relurpify is LLM agnostic.

The Euclo runtime must therefore support lower-end models as a first-class target, not as a degraded afterthought.

### 14.1 Optimization Goal

Out-of-the-box Euclo should optimize for:

- reliable execution
- grounded knowledge surfacing
- continuity over verbosity
- correct use of Archaeo as restoration substrate

### 14.2 Fresh Reasoning Dependence

Euclo should avoid unnecessary dependence on fresh model calls during long-running execution.

The exact threshold may vary by mode, but in archaeology-backed long-running execution:

- context management should depend on Archaeo and compiled execution state
- restoration should not require recreating the entire plan from scratch
- execution should prefer deterministic continuation where safe

### 14.3 Knowledge Surfacing

When Archaeo context exists, Euclo should bias toward:

- pattern-backed outputs
- tension-aware reasoning
- provenance-grounded explanations
- explicit possibility surfacing

rather than unsupported freeform prose summaries.

## 15. Stop vs Continue Semantics

Long-running implementation should continue by default.

The expected behavior is:

- keep executing
- collect deferred execution issues
- restore context as needed
- surface unresolved concerns as artifacts

### 15.1 Hard Interruptions

Only a narrow set of events should stop execution:

- explicit user cancel
- irrecoverable sandbox/security denial
- irrecoverable runtime corruption
- inability to restore safe continuation

Everything else should prefer:

- defer
- checkpoint
- restore
- continue

## 16. Benchmark Scenarios

Eucloâ€™s effectiveness should be evaluated against at least these scenarios:

- failing test to fix
- multi-step implementation against a living plan
- compatibility-preserving refactor
- long-running migration or large execution workflow
- resume after context compaction and restoration

These should become product-quality benchmark scenarios, not just ad hoc tests.

## 17. Required Artifact Families

The full-product Euclo runtime should expose at minimum the following artifact families:

- `compiled_execution`
- `execution_status`
- `verification_status`
- `deferred_execution_issues`
- `context_compaction`
- `checkpoint_summary`
- `final_report`

These artifacts must be stable enough that multiple UX layers can consume them consistently.

## 18. Relationship to Relurpish

This specification does not define Relurpish presentation.

However, it does require Euclo to expose enough runtime state that Relurpish can later surface:

- deferred occurrence notifications
- context compaction and restoration notifications
- execution-aware finalize state
- workflow-backed long-running continuity

Those are downstream UX responsibilities, not Euclo runtime responsibilities.

## 19. Relationship to Archaeo GraphQL

This specification assumes Archaeo continues to expose external archaeology state through surfaces such as `/app/archaeo-graphql-server`.

That server is a transport/control-plane adapter over Archaeo, not Eucloâ€™s execution owner.

Euclo may consume Archaeo state through bindings, local runtime access, or remote transport, but its runtime semantics remain defined here.

## 20. Non-Goals

This specification does not define:

- a specific TUI layout
- pane/tab behavior
- GraphQL schema changes for external archaeology UX
- a minimum viable rollout plan

This is a full-product target specification for Eucloâ€™s runtime contract.

## 21. Implementation Directions

The likely implementation tracks implied by this specification are:

- define the compiled execution artifact schema and persistence
- define the deferred execution issue schema, persistence, and workspace artifact emission
- add explicit context compaction and automatic restore state handling
- strengthen execution-state persistence around workflow-backed runs
- formalize relurpic routine selection as preset-plus-step execution logic
- add benchmark coverage for long-running continuity and lower-end model operation

## 22. Decision Summary

The key product decisions captured here are:

- Euclo is the UX-agnostic coding runtime
- Archaeo is the living memory/provenance/planning substrate
- workflow-backed execution is the default for substantial coding work
- mode/phase remains the primary execution control model
- compiled execution state is first-class
- deferred execution issues are first-class Euclo artifacts
- automatic context restore is a hard guarantee for long-running implementation tasks
- relurpic capabilities are reusable execution routines and presets
- Euclo must be reliable and knowledge-capable even on lower-end LLMs
