# Euclo UnitOfWork Runtime Object Specification

Status

Proposed runtime object specification.

Scope

This document defines `UnitOfWork` as an explicit Euclo runtime object.

It extends:

- [5.md](/home/lex/Public/Relurpify/docs/research/5.md)
- [6.md](/home/lex/Public/Relurpify/docs/research/6.md)
- [7.md](/home/lex/Public/Relurpify/docs/research/7.md)

The purpose of this object is to give Euclo a stronger internal execution model than:

- mode alone
- profile alone
- scattered runtime state

## 1. Why UnitOfWork Exists

Euclo should behave like a toolkit that collects context in order to execute its current work.

That work is not defined only by:

- a prompt
- a profile
- a plan

It is defined by a runtime assembly of:

- agent execution intent
- relurpic execution behavior
- skill usage
- tool capability usage
- context gathered to support the behavior
- optional living-plan binding

`UnitOfWork` is the explicit runtime object that captures that assembly.

## 2. Definition

`UnitOfWork` is a Euclo-owned runtime object describing the current execution bundle for one coherent piece of work.

A `UnitOfWork` should answer:

- what Euclo is trying to do
- how it intends to do it
- what context it gathered
- what relurpic routines are bound
- what tools and skills are in scope
- what verification and deferral policies apply
- whether the work is plan-backed

## 3. Ownership

`UnitOfWork` is owned by Euclo.

It may reference:

- Archaeo plan/provenance state
- relurpic routines
- framework skills
- tool capability policies

But it is not an Archaeo object and should not require Archaeo changes.

## 4. Role In The Runtime

Euclo should move toward:

- selecting or deriving a `UnitOfWork`
- gathering context for that `UnitOfWork`
- executing through that `UnitOfWork`
- persisting enough of that `UnitOfWork` for resume, reporting, and restore

This is a deeper abstraction than profile selection.

Profiles may still exist, but they should act more like inputs or presets into `UnitOfWork` assembly rather than the final execution model.

## 5. Core Structure

Suggested high-level structure:

```go
type UnitOfWork struct {
    ID string

    WorkflowID  string
    RunID       string
    ExecutionID string

    ModeID              string
    ObjectiveKind       string
    BehaviorFamily      string
    ContextStrategyID   string
    VerificationPolicy  string
    DeferralPolicy      string

    PlanBinding         *UnitOfWorkPlanBinding
    ContextBundle       UnitOfWorkContextBundle
    RoutineBindings     []UnitOfWorkRoutineBinding
    SkillBindings       []UnitOfWorkSkillBinding
    ToolBindings        []UnitOfWorkToolBinding
    CapabilityBindings  []UnitOfWorkCapabilityBinding

    Status              UnitOfWorkStatus
    ResultClass         string
    DeferredIssueIDs    []string
    CreatedAt           time.Time
    UpdatedAt           time.Time
}
```

The exact Go shape may differ, but the runtime semantics should be preserved.

## 6. UnitOfWork Fields

### 6.1 Identity

Required identity fields:

- `id`
- `workflow_id`
- `run_id`
- `execution_id`

These allow the object to participate in persistence, restore, and reporting.

### 6.2 Execution Intent

Required execution-intent fields:

- `mode_id`
- `objective_kind`
- `behavior_family`

These fields answer:

- what category of Euclo mode is active
- what the work is meant to achieve
- what relurpic-style execution behavior is leading the work

### 6.3 Policy Fields

Required policy fields:

- `context_strategy_id`
- `verification_policy`
- `deferral_policy`
- `checkpoint_policy`

These define how the unit of work behaves under execution pressure.

### 6.4 Status Fields

Required status fields:

- `status`
- `result_class`
- `deferred_issue_ids`

These define the current and final state of the work unit.

## 7. Plan Binding

Not every `UnitOfWork` needs a living plan binding.

However, `plan` mode and long-running execution paths should support one explicitly.

Suggested structure:

```go
type UnitOfWorkPlanBinding struct {
    WorkflowID         string
    PlanID             string
    PlanVersion        int
    ActiveStepID       string
    StepIDs            []string
    IsPlanBacked       bool
    IsLongRunning      bool
    ArchaeoRefs        map[string][]string
}
```

### 7.1 Important Rule

The presence of `PlanBinding` should indicate that the unit of work is executing against the single-plan contract defined in [6.md](/home/lex/Public/Relurpify/docs/research/6.md).

### 7.2 Optionality

Direct or lighter-weight work units may omit `PlanBinding` entirely.

This preserves Euclo’s toolkit nature while still allowing heavy plan-backed execution.

## 8. Context Bundle

Context is central to Euclo.

`UnitOfWork` should carry an explicit context bundle rather than relying on implicit state scattered across runtime keys.

Suggested structure:

```go
type UnitOfWorkContextBundle struct {
    Sources              []UnitOfWorkContextSource
    WorkspacePaths       []string
    RetrievalRefs        []string
    ArtifactKinds        []string
    PatternRefs          []string
    TensionRefs          []string
    ProvenanceRefs       []string
    LearningRefs         []string
    ContextBudgetClass   string
    CompactionEligible   bool
    RestoreRequired      bool
}
```

### 8.1 Purpose

This bundle describes what context was gathered to support the current unit of work.

It is especially important for:

- lower-end model operation
- context compaction
- restore
- reporting why a given execution path was chosen

## 9. Routine Bindings

Relurpic capabilities should be bound into the `UnitOfWork` explicitly.

Suggested structure:

```go
type UnitOfWorkRoutineBinding struct {
    RoutineID       string
    Family          string
    Reason          string
    Priority        int
    Required        bool
}
```

### 9.1 Purpose

This makes execution behavior explicit.

Examples:

- `gap_analysis`
- `verification_repair`
- `scope_expansion_assessment`
- `stale_assumption_detection`
- `tension_assessment`

### 9.2 Why This Matters

Euclo should not look like it is merely “running a profile.”

It should look like it assembled a work unit with concrete behavior bindings.

## 10. Skill Bindings

Because unit of work intersects with skills, this should be explicit too.

Suggested structure:

```go
type UnitOfWorkSkillBinding struct {
    SkillID      string
    Reason       string
    Required     bool
}
```

This allows Euclo to express that a work unit is not just a plan/tool execution path, but also a knowledge/configuration-guided path.

## 11. Tool And Capability Bindings

Tool and capability scope should also be explicit inside the `UnitOfWork`.

Suggested structures:

```go
type UnitOfWorkToolBinding struct {
    ToolID        string
    Allowed       bool
    Reason        string
}

type UnitOfWorkCapabilityBinding struct {
    CapabilityID  string
    Family        string
    Required      bool
}
```

This gives Euclo a way to express:

- which tools were intentionally admitted
- which capabilities are part of the execution assembly
- why they were selected

## 12. Objective Kind

`objective_kind` should be a first-class discriminator.

Suggested examples:

- `direct_execution`
- `investigation`
- `verification_repair`
- `plan_execution`
- `review`
- `refactor`
- `migration`

This helps Euclo distinguish what type of work is being assembled even when mode alone is insufficient.

## 13. Behavior Family

`behavior_family` should describe the primary relurpic-style execution behavior leading the work unit.

Suggested initial values:

- `gap_analysis`
- `verification_repair`
- `scope_expansion_assessment`
- `stale_assumption_detection`
- `tension_assessment`
- `direct_change_execution`

Mode and behavior family should remain distinct.

Mode is broader.

Behavior family is the narrower execution behavior bias for this specific unit of work.

## 14. Status Model

Suggested `UnitOfWorkStatus` values:

- `assembling`
- `ready`
- `executing`
- `verifying`
- `compacted`
- `restoring`
- `completed`
- `completed_with_deferrals`
- `blocked`
- `failed`
- `canceled`

This can align with Euclo execution status artifacts, but `UnitOfWork` should remain the owner of its own current runtime meaning.

## 15. Relationship To Profiles

Profiles should not disappear immediately, but their role should narrow.

Profiles should become:

- presets
- hints
- compatibility adapters

for assembling a `UnitOfWork`.

They should no longer be the deepest or most expressive execution abstraction.

## 16. Relationship To Compiled Execution

`CompiledExecution` and `UnitOfWork` are related but not identical.

### 16.1 UnitOfWork

Represents the active execution assembly.

It is the runtime-facing object.

### 16.2 CompiledExecution

Represents the durable continuity and resume contract.

It is the persistence-facing object.

### 16.3 Recommended Relationship

Euclo should be able to:

- assemble a `UnitOfWork`
- derive a `CompiledExecution` snapshot from it
- restore a `UnitOfWork` from persisted compiled execution plus current state

That gives a cleaner runtime/persistence split.

## 17. Relationship To Deferrals

Deferred execution issues should be attached to the `UnitOfWork`.

This allows a work unit to clearly answer:

- what concerns were discovered
- whether they were step-local or run-level
- whether final classification is `completed_with_deferrals`

## 18. Direct vs Plan-Backed Work

`UnitOfWork` should support both:

- lighter direct execution
- heavy plan-backed long-running execution

This is essential because Euclo is a toolkit, not only a plan executor.

### 18.1 Direct Work

Characteristics:

- lighter context bundle
- no plan binding required
- fewer routine bindings
- faster execution path

### 18.2 Plan-Backed Work

Characteristics:

- large context bundle
- explicit living-plan binding
- long-running continuity
- compiled execution persistence
- stronger deferral semantics
- restore expectations

## 19. Package Direction

Likely homes:

- `named/euclo/euclotypes`
- or new `named/euclo/workunit`
- or new `named/euclo/executionstate`

My bias is to keep `UnitOfWork` in a dedicated Euclo runtime package rather than burying it only in general type exports.

Suggested package candidates:

- `named/euclo/workunit`
- `named/euclo/runtime/workunit.go`

## 20. Implementation Direction

The likely implementation sequence is:

1. define `UnitOfWork` types
2. derive `UnitOfWork` from current mode/profile/classification/runtime state
3. bind relurpic routine families explicitly
4. bind skills/tools/capabilities explicitly
5. make finalization and restore operate in terms of `UnitOfWork`
6. keep profiles as compatibility presets during migration

## 21. Benefits

Making `UnitOfWork` explicit gives Euclo:

- clearer runtime ownership
- better introspection
- a stronger restore model
- cleaner integration with relurpic routines
- a better abstraction than “mode + profile + implicit state”
- a way to stay toolkit-like while still supporting heavy long-context plan execution

## 22. Summary

The key decision is:

`UnitOfWork` should become a first-class Euclo runtime object.

It represents the assembled execution bundle for the current work, including:

- execution intent
- context gathered
- relurpic behavior bindings
- skill/tool/capability scope
- optional living-plan binding
- verification and deferral policies
- final status and result classification

This gives Euclo a stronger, more self-owning execution model while preserving its identity as a toolkit that gathers context in order to execute coding work.
