# Euclo Implementation Strategy Without Archaeo Modification

Status

Proposed implementation strategy.

Scope

This document defines how to implement the runtime contracts in:

- [5.md](/home/lex/Public/Relurpify/docs/research/5.md)
- [6.md](/home/lex/Public/Relurpify/docs/research/6.md)

under the explicit constraint that:

- `archaeo` should not need modification

This is a key architectural constraint.

Euclo must implement its stronger execution/runtime model by using existing Archaeo services, bindings, requests, records, projections, and references as a fixed substrate.

## 1. Constraint

The target architecture is:

- Archaeo requests and evaluates
- Euclo executes
- relurpic capabilities reason

But Archaeo is treated as stable for this work.

That means:

- no new Archaeo domain types should be required
- no new Archaeo persistence schema should be assumed
- no new Archaeo execution-specific semantics should be pushed down
- no Euclo-specific runtime concepts should require Archaeo package changes

The implementation must be Archaeo-first in integration, but Archaeo-stable in change surface.

## 2. Consequence Of The Constraint

All of the following become Euclo-owned concerns:

- deferred execution issue types
- deferred execution issue taxonomy
- deferred execution workspace artifacts
- execution result classes such as `completed_with_deferrals`
- compiled execution persistence details
- execution-status persistence details
- context compaction and restore state artifacts
- run-level indexes over execution issues

Archaeo may be referenced by these artifacts, but it does not own them.

## 3. Fixed Archaeo Surface

For the purposes of this implementation strategy, Archaeo should be consumed only through existing surfaces such as:

- direct services in `/archaeo/*`
- existing higher-level bindings
- existing projections
- existing request lifecycle services
- existing provenance, tension, learning, plan, and phase references

`/app/archaeo-graphql-server` remains an optional transport surface and should not be the default runtime integration path for in-process Euclo or relurpic execution.

## 4. Ownership Map

### 4.1 Euclo Owns

`/named/euclo` should own:

- deferred execution issue model
- compiled execution model
- execution result classes
- execution-status model
- workspace artifact emission for Euclo runtime artifacts
- Euclo-to-Archaeo reference mapping helpers
- execution restore metadata
- run summary assembly including deferral state

### 4.2 Relurpic Owns

`/agents/relurpic` should own:

- behavior-oriented reasoning routines
- execution-time reasoning helpers used by Euclo
- direct reads/writes through existing Archaeo services and bindings where needed

Relurpic should not require new Archaeo domain expansion in order to support Euclo’s stronger runtime.

### 4.3 Framework Owns

`/framework` should only be touched if shared substrate support is needed, such as:

- generic artifact persistence helpers
- context lifecycle storage helpers
- reusable execution status infrastructure

Framework changes, if any, should remain generic and not Euclo-specific.

## 5. Euclo-Owned Types

The following types should be introduced or formalized in Euclo-owned packages.

### 5.1 CompiledExecution

Suggested home:

- `named/euclo/euclotypes`
- or new `named/euclo/executionstate`

This type should be fully Euclo-owned.

It may include Archaeo references, but it is not an Archaeo type.

### 5.2 DeferredExecutionIssue

Suggested home:

- `named/euclo/euclotypes`
- or new `named/euclo/executionstate`

This type must be Euclo-owned because:

- it is execution-specific
- it is run-scoped
- it is not an Archaeo deferred draft
- it must exist even if Archaeo remains unchanged

### 5.3 ExecutionResultClass

Suggested values:

- `completed`
- `completed_with_deferrals`
- `blocked`
- `failed`
- `canceled`
- `restore_failed`

This classification is Euclo runtime semantics and should remain in Euclo.

### 5.4 ExecutionStatus

Euclo should own a stable execution status type for:

- current state
- persistence
- reporting
- restore

Archaeo phase state may still be referenced, but Euclo must not rely on Archaeo owning its runtime result classes.

## 6. Euclo Artifact Strategy

Because Archaeo should not change, Euclo must persist its runtime concerns using its own artifact layer.

### 6.1 Artifact Families

At minimum Euclo should persist:

- `euclo.compiled_execution`
- `euclo.execution_status`
- `euclo.deferred_execution_issue`
- `euclo.context_compaction`
- `euclo.restore_summary`
- `euclo.final_report`

The exact artifact kinds can remain Euclo-local even when they include Archaeo references.

### 6.2 Workflow Artifact Storage

Euclo should persist these through existing workflow artifact mechanisms already available to the runtime.

No new Archaeo storage contract should be necessary.

### 6.3 Workspace Artifact Emission

For end-user review, Euclo should emit selected runtime artifacts into workspace-visible files, especially:

- deferred execution issue markdown files with YAML frontmatter

This emission should be implemented in Euclo rather than Archaeo.

## 7. Euclo To Archaeo Reference Mapping

Euclo should adopt a reference-mapping layer rather than ask Archaeo to understand Euclo-native runtime semantics.

### 7.1 Purpose

Reference mapping allows Euclo artifacts to point to:

- tensions
- provenance refs
- requests
- learning interactions
- phase transitions
- plan versions

without requiring Archaeo to understand Euclo deferral logic.

### 7.2 Mapping Principle

Archaeo IDs and records are treated as external semantic references embedded into Euclo artifacts.

Euclo is responsible for:

- capturing the references
- preserving them
- surfacing them later

Archaeo remains responsible only for its own records.

### 7.3 Suggested Helper Layer

Euclo should have a small helper layer for:

- collecting Archaeo refs from runtime state
- normalizing them into Euclo issue artifacts
- reconstructing them during restore/final reporting

Suggested homes:

- `named/euclo/runtime`
- or `named/euclo/executionstate`

## 8. Deferred Execution Issues Without Archaeo Changes

The deferred issue model should be implemented entirely in Euclo.

### 8.1 Input Sources

Deferred issue generation may consume:

- current step data
- compiled execution state
- verification evidence
- relurpic reasoning outputs
- Archaeo references already available from services or state

### 8.2 Output

Deferred issue generation must produce:

- Euclo runtime object
- workflow artifact
- markdown workspace artifact
- run-level issue index update

### 8.3 No New Archaeo Objects Required

If later product work wants deeper Archaeo alignment, that can happen separately.

For this implementation strategy, Euclo must work fully with no Archaeo modification.

## 9. Result Class Strategy

Because `completed_with_deferrals` is a Euclo-specific top-level result class, it should be implemented in Euclo finalization/reporting rather than in Archaeo.

### 9.1 Finalization Responsibility

Euclo finalization should:

- inspect deferred issue state
- compute the final result class
- persist execution status artifact
- assemble the final report

### 9.2 Compatibility

This can coexist with existing success/failure booleans for compatibility, but the richer result class should be the canonical runtime classification.

## 10. Context Restore Without Archaeo Changes

Automatic restore remains a hard guarantee for long-running implementation tasks.

With Archaeo unchanged, Euclo must implement restore coordination by using:

- existing Archaeo projections
- existing plan and provenance state
- existing workflow artifacts
- Euclo-owned compiled execution artifacts
- Euclo-owned deferral indexes

### 10.1 Restore Logic Ownership

Restore orchestration should live in Euclo.

Archaeo remains the source of living semantic context, but Euclo decides how to reconstruct execution continuity.

### 10.2 Restore Summary

Euclo should persist a restore summary artifact so that:

- future restores do not need to re-infer everything from scratch
- downstream UX layers can report what happened

## 11. Relurpic Routine Wiring Without Archaeo Changes

Relurpic behavior-oriented routines should be wired against existing Archaeo services and bindings directly.

### 11.1 Target Pattern

The target pattern is:

- Euclo invokes a behavior-oriented relurpic routine
- the routine reads existing Archaeo state directly
- the routine returns reasoning output to Euclo
- Euclo decides whether to continue, defer, checkpoint, or classify results

### 11.2 Avoided Pattern

The avoided pattern is:

- expanding Archaeo to host new Euclo execution behavior
- requiring GraphQL in-process for local execution
- coupling relurpic routines to raw provider/store wiring when a safer Archaeo-facing binding already exists

### 11.3 Likely Cleanup Direction

If relurpic provider behavior still uses raw handler-plus-store composition as a primary path, it should be migrated toward:

- Archaeo-first reads and fulfillment boundaries
- Euclo-owned execution decision logic
- behavior-family contracts rather than ad hoc handler coupling

## 12. Package-Level Likely Changes

### 12.1 Named Euclo

Likely areas:

- `named/euclo/euclotypes`
- `named/euclo/runtime`
- `named/euclo/agent.go`
- `named/euclo/orchestrate`
- possibly new `named/euclo/executionstate`

### 12.2 Agents Relurpic

Likely areas:

- `agents/relurpic/*`

Focus:

- behavior-family organization
- Archaeo-first integration cleanup
- execution reasoning contract cleanup

### 12.3 Framework

Only if necessary:

- generic artifact persistence helpers
- generic context lifecycle helpers

### 12.4 Archaeo

No required modifications.

## 13. Migration Strategy

The recommended migration strategy is additive and Euclo-owned.

### 13.1 Phase 1

Introduce Euclo-owned artifact types and result classes:

- compiled execution
- execution status
- deferred execution issue
- result classification

### 13.2 Phase 2

Add workspace artifact emission for deferred issues.

### 13.3 Phase 3

Bind restore and finalization logic to Euclo-owned execution artifacts.

### 13.4 Phase 4

Refactor relurpic behavior families around execution behaviors and existing Archaeo integration surfaces.

### 13.5 Phase 5

Reduce remaining architectural ambiguity in Euclo runtime ownership, including any oversized delegate role from generic agent implementations.

## 14. What This Constraint Prevents

This strategy intentionally avoids:

- treating Archaeo as the owner of Euclo runtime semantics
- introducing Euclo-specific Archaeo records
- making GraphQL the in-process integration dependency
- coupling execution continuity to future Archaeo schema expansion

## 15. Benefits

This strategy has several benefits:

- respects the stability of Archaeo
- keeps the Euclo/Archaeo boundary cleaner
- makes Euclo’s runtime identity more explicit
- reduces architectural pressure to encode execution semantics into planning infrastructure
- supports incremental implementation without large semantic churn in Archaeo

## 16. Main Engineering Implication

The main implication is straightforward:

if the stronger runtime model is the goal, then Euclo must become more self-owning.

Archaeo can remain the semantic substrate, but Euclo must take full ownership of:

- its runtime artifacts
- its result classes
- its deferral semantics
- its restore semantics
- its execution-state persistence

## 17. Summary

Under the constraint that Archaeo should not need modification:

- Euclo becomes the owner of the new execution/runtime semantics
- relurpic routines become behavior-oriented reasoning units using existing Archaeo surfaces
- Archaeo remains the planning/provenance/evaluation substrate
- GraphQL remains an optional external transport

This is the cleanest implementation path that preserves the intended architecture without destabilizing Archaeo.
